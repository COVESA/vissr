§1 Overview

VISSv4.0 extends vissr with native support for the COVESA VDM signal-model format. Where earlier releases required a compiled VSS binary tree (loaded via the viss.him forest manifest), v4.0 allows operators to supply a directory of raw .graphql SDL files instead. The server parses them at startup, builds the equivalent utils.Node_t signal trees in memory, and registers them in the HIM forest — with no binary pre-compilation step required.

Key additions:

  • vdmloader package — pure-Go SDL parser (uses github.com/vektah/gqlparser/v2).
  • --vdm <dir> server flag — alternative to --him; loads every *.graphql file in dir.
  • @viss_service directive — marks a VDM field as a service procedure entry point, enabling VISSv3.2 invoke/monitor semantics on VDM-defined signals.

§2 VDM Background

The Vehicle Data Model (VDM) is a COVESA initiative that expresses the VSS signal catalog in GraphQL Schema Definition Language (SDL) rather than YAML. The SDL is authored using the s2dm (Simplified Semantic Data Modeling) methodology and toolchain, which validates structural and semantic constraints beyond what the GraphQL type system alone enforces.

A VDM SDL file contains:

  • Object types annotated with @vspec(element: BRANCH, fqn: "Dot.Path") — tree branch nodes.
  • Field definitions annotated with @vspec(element: SENSOR|ACTUATOR|ATTRIBUTE, fqn: "Dot.Leaf.Path") — signal leaf nodes.
  • Custom scalars (Int8, UInt8, etc.) that map to VSS datatypes.
  • Optional @range(min, max) for numeric bounds.

VDM has no notion of mutations or procedures — it is a pure signal/data model. The @viss_service directive described in §8 is a vissr extension that does not yet exist in upstream VDM.

§3 Supported SDL Directives

3.1 @vspec

directive @vspec(
  element: VspecElement!   # node kind — see §4
  fqn:     String!         # fully-qualified dot-separated signal path
  description: String      # human-readable description
  comment:     String      # implementation note (ignored at runtime)
  deprecation: String      # deprecation notice (ignored at runtime)
) on OBJECT | FIELD_DEFINITION

3.2 @range

directive @range(min: Float, max: Float) on FIELD_DEFINITION

Populates Node_t.Min and Node_t.Max as decimal strings.

3.3 @viss_service (v4.0 extension)

directive @viss_service on FIELD_DEFINITION

When present on a field definition, the loader creates a PROCEDURE node instead of the element type declared in @vspec. This enables VISSv3.2 service semantics (invoke, monitor, cancel, discover) on the field. See §8 for details.

§6 Tree Reconstruction Algorithm

The @vspec(fqn: …) attribute is the authoritative source of tree structure. The GraphQL type hierarchy is not used for parent-child relationships; only the FQN strings are.

  1. Parse the combined preamble + SDL with gqlparser.LoadSchema.
  2. Walk every type and field definition. For each @vspec-annotated node, record fqn → Node_t in a flat map.
  3. Sort all FQN strings lexicographically so parents always appear before children.
  4. For each non-root FQN, compute the parent FQN by stripping the last dot-segment. If the parent is missing from the map, synthesise an intermediate BRANCH node and insert it.
  5. Call appendChild(parent, child) to link the node and maintain the Children counter.
  6. Collect all root nodes (FQN contains no .) and return them.

Implementations MUST preserve the invariant node.Children == uint8(len(node.Child)) after every appendChild call.

§7 Instance Tags (multi-instance)

VDM represents multi-instance nodes (e.g., seat rows × sides) using instance-tag types and dimension enums. A type named {Base}_InstanceTag bearing @instanceTag is a multi-instance template: the loader expands it into one subtree per combination of dimension values and attaches those subtrees under the base FQN branch.

7.1 SDL pattern

type VehicleCabinSeat @vspec(element: BRANCH, fqn: "Vehicle.Cabin.Seat") {
  IsOccupied: Boolean @vspec(element: SENSOR, fqn: "Vehicle.Cabin.Seat.IsOccupied")
  Position:   UInt8   @vspec(element: ACTUATOR, fqn: "Vehicle.Cabin.Seat.Position") @range(min: 0, max: 100)
}

type VehicleCabinSeat_InstanceTag @instanceTag @vspec(element: BRANCH, fqn: "Vehicle.Cabin.Seat") {
  dimension1: VehicleCabinSeat_InstanceTag_Dimension1
  dimension2: VehicleCabinSeat_InstanceTag_Dimension2
}

enum VehicleCabinSeat_InstanceTag_Dimension1 {
  ROW1 @vspec(originalName: "Row1")
  ROW2 @vspec(originalName: "Row2")
}

enum VehicleCabinSeat_InstanceTag_Dimension2 {
  DRIVER_SIDE    @vspec(originalName: "DriverSide")
  PASSENGER_SIDE @vspec(originalName: "PassengerSide")
}

7.2 Expansion result

The loader expands the above into the following tree (all leaf signals cloned into every instance):

Vehicle.Cabin.Seat (BRANCH)
  ├── Row1 (BRANCH)
  │    ├── DriverSide (BRANCH)
  │    │    ├── IsOccupied (SENSOR, bool)
  │    │    └── Position   (ACTUATOR, uint8, range 0–100)
  │    └── PassengerSide (BRANCH)
  │         ├── IsOccupied (SENSOR, bool)
  │         └── Position   (ACTUATOR, uint8, range 0–100)
  └── Row2 (BRANCH)
       ├── DriverSide (BRANCH)
       │    ├── IsOccupied (SENSOR, bool)
       │    └── Position   (ACTUATOR, uint8, range 0–100)
       └── PassengerSide (BRANCH)
            ├── IsOccupied (SENSOR, bool)
            └── Position   (ACTUATOR, uint8, range 0–100)

7.3 Naming and SCREAMING_SNAKE_CASE fallback

Each enum value's expanded path segment comes from @vspec(originalName: "..."). If this argument is absent the loader converts the enum value name from SCREAMING_SNAKE_CASE to PascalCase: DRIVER_SIDE → DriverSide, ROW1 → Row1.

7.4 Dimensions

Up to N dimensions are supported (VDM currently uses 1 or 2). Dimension fields must be named dimension1, dimension2, … in order on the _InstanceTag type. Instances are generated in lexicographic order of dimension value combinations.

A conforming implementation MUST deep-clone the full signal subtree of the base FQN into every generated instance, preserving all field values (datatype, min, max, unit, description) and maintaining independent *Node_t pointers per instance so mutations to one instance do not affect others.

§8 Unit, Default Value, and Allowed Values

8.1 Unit

A signal's physical unit is declared with the @unit(value:) directive or the unit argument of @vspec. The value may be a plain string ("km/h") or a SCREAMING_SNAKE_CASE enum name that the loader maps to a standard string:

Speed: Float @vspec(element: SENSOR, fqn: "Vehicle.Speed") @unit(value: "KM_PER_HOUR")
# → Node_t.Unit = "km/h"

The full normalisation table is defined in vdmloader/vdmloader.go (normalizeUnit). Values not in the table are passed through unchanged.

8.2 Default value

A signal's initial value at startup is declared with @defaultValue(value: "…"):

IsEnabled: Boolean @vspec(element: ACTUATOR, fqn: "…") @defaultValue(value: "false")
# → Node_t.DefaultValue = "false"

8.3 Allowed values (enum signals)

When a signal field is typed as a user-defined GraphQL enum, the loader populates Node_t.AllowedDef with the enum values. @vspec(originalName:) on each enum value controls the string stored in AllowedDef; it falls back to SCREAMING_SNAKE_CASE → PascalCase conversion.

enum GearPosition {
  PARK    @vspec(originalName: "Park")
  REVERSE @vspec(originalName: "Reverse")
  DRIVE   @vspec(originalName: "Drive")
}
type Vehicle … {
  CurrentGear: GearPosition
    @vspec(element: SENSOR, fqn: "Vehicle.CurrentGear")
    @defaultValue(value: "Park")
}
# → Node_t.AllowedDef = ["Park","Reverse","Drive"]
# → Node_t.Allowed = 3
# → Node_t.DefaultValue = "Park"

Internal VDM enums (VspecElement, *_InstanceTag_Dimension*) are never exposed as allowed values.

§9 @viss_service Extension

VDM itself has no concept of procedures or mutations. The @viss_service directive is a vissr-specific extension that service developers add to their VDM field definitions to expose them as callable service procedures over VISSv3.2 invoke/monitor channels.

8.1 Declaring a service procedure

type VehicleSeating @vspec(element: BRANCH, fqn: "VehicleSeating", description: "Seating service") {
  MoveSeat: Boolean
    @vspec(element: SENSOR, fqn: "VehicleSeating.MoveSeat", description: "Adjust a seat position")
    @viss_service
}

The loader will create a PROCEDURE node at VehicleSeating.MoveSeat. The vissr service manager (vissServiceMgr) treats PROCEDURE nodes exactly as it does procedure nodes loaded from a .Service HIM tree.

8.2 Interaction with VISSv3.2 service protocol

Once the VDM tree is registered, clients can interact with @viss_service fields using the standard VISSv3.2 actions: invoke, monitor, cancel, and discover.

Note: The @viss_service directive is not part of the upstream VDM or s2dm specification. It is intended as a vissr-specific overlay that service authors add to their own VDM extension files without modifying the base VDM schema.

§10 Server Integration (--vdm flag)

9.1 Command-line flag

vissv2server --vdm <dir>

When --vdm is supplied, the server skips viss.him loading and instead calls vdmloader.LoadDir(dir) at startup. All *.graphql files in dir are parsed in lexicographic order. Each distinct root FQN (e.g., Vehicle) becomes a separate tree registered in the HIM forest via utils.RegisterServiceTree.

9.2 Mutual exclusivity

--vdm and the default viss.him loading are mutually exclusive. If both a VDM directory and a viss.him file are relevant, run the server with --vdm (the viss.him path will not be read).

9.3 Multiple SDL files

All *.graphql files in the directory are loaded. If two files define a root with the same FQN (e.g., both export a Vehicle root), the second registration is silently skipped and a warning is logged. Only the first file's tree is used.

10.4 Web dashboard (--web-addr)

The optional embedded web dashboard is enabled with:

vissv2server --vdm <dir> --web-addr :8090

The dashboard serves an interactive signal-tree explorer at http://localhost:8090. JSON API endpoints:

PathDescription
GET /api/forestArray of {rootName, domain, version} objects for every loaded tree.
GET /api/tree/{rootName}Full node tree as JSON with name, nodeType, datatype, unit, min, max, default, allowed, children.

The dashboard is disabled by default (empty --web-addr) and has no runtime dependencies beyond Go's standard library.

§11 HIM / VDM Interoperability

v4.0 VDM trees and v3.x HIM trees use the same internal utils.Node_t representation. A server instance can only load one or the other at startup, but the VISS protocol layer cannot distinguish between the two at runtime — signal paths, subscriptions, and discovery work identically regardless of how the tree was loaded.

§12 vdminfo Inspection Tool

The tools/vdminfo package is a standalone CLI for inspecting VDM SDL files without starting the full server:

go run ./tools/vdminfo ./my-vdm-files/

Useful in CI to diff tree output between VDM revisions and to verify that instance tag expansion produces the expected signal paths.

§13 CI Validation with s2dm

The s2dm Python toolchain validates that VDM SDL files are structurally and semantically correct (naming conventions, forbidden patterns, required directive presence, etc.). vissr does not depend on s2dm at runtime; s2dm is a CI gate only.

To add s2dm validation to a vissr CI pipeline:

# In .github/workflows/test.yml, before Go tests:
- name: Validate VDM SDL (s2dm)
  run: |
    pip install uv
    uv run poe validate --input path/to/vdm-files/

s2dm validates that every type has a valid @vspec annotation, that FQN strings are well-formed, and that COVESA naming conventions are respected. The loader's own tests cover parse correctness; s2dm covers semantic compliance with the VDM authoring rules.

§14 Conformance

A conforming v4.0 VDM loader implementation MUST:

  1. Accept the preamble in Appendix A as valid SDL without error.
  2. Reconstruct the tree hierarchy from @vspec(fqn: …) values, not from the GraphQL type graph.
  3. Map all GraphQL types listed in §5 to the correct VSS datatype string.
  4. Populate Node_t.Min and Node_t.Max from @range as decimal strings without trailing zeros.
  5. Override NodeType to PROCEDURE for any field bearing @viss_service.
  6. Synthesise intermediate BRANCH nodes for any FQN parent not explicitly declared in the SDL.
  7. Maintain Node_t.Children == uint8(len(Node_t.Child)) at all times.
  8. Expand every *_InstanceTag type bearing @instanceTag into one subtree per dimension-value combination, deep-cloning the base signal subtree into each instance with independent *Node_t pointers.
  9. Derive instance path segments from @vspec(originalName:) on enum values, falling back to SCREAMING_SNAKE_CASE → PascalCase conversion when the argument is absent.
  10. Reject SDL that contains no @vspec-annotated type (return an error).
  11. Reject SDL that contains no root-level BRANCH type (return an error).

Appendix B — Example VDM File

The following vehicle.graphql file demonstrates a minimal but representative VDM signal model that the v4.0 loader can consume directly.

# vehicle.graphql — minimal VDM SDL example for vissr v4.0

type Vehicle @vspec(element: BRANCH, fqn: "Vehicle", description: "Top-level vehicle object") {
  Speed: Float         @vspec(element: SENSOR,    fqn: "Vehicle.Speed",    description: "Current speed in km/h") @range(min: 0, max: 250)
  ADAS:  VehicleADAS   @vspec(element: BRANCH,    fqn: "Vehicle.ADAS")
  Cabin: VehicleCabin  @vspec(element: BRANCH,    fqn: "Vehicle.Cabin")
}

type VehicleADAS @vspec(element: BRANCH, fqn: "Vehicle.ADAS", description: "Advanced driver-assistance systems") {
  ActiveAutonomyLevel: String  @vspec(element: ATTRIBUTE, fqn: "Vehicle.ADAS.ActiveAutonomyLevel", description: "Current autonomy level")
  ABS: VehicleADASABS          @vspec(element: BRANCH,    fqn: "Vehicle.ADAS.ABS")
}

type VehicleADASABS @vspec(element: BRANCH, fqn: "Vehicle.ADAS.ABS", description: "Antilock braking system") {
  IsEnabled: Boolean @vspec(element: ACTUATOR, fqn: "Vehicle.ADAS.ABS.IsEnabled", description: "Enable or disable ABS")
  IsActive:  Boolean @vspec(element: SENSOR,   fqn: "Vehicle.ADAS.ABS.IsActive",  description: "ABS is currently active")
}

type VehicleCabin @vspec(element: BRANCH, fqn: "Vehicle.Cabin", description: "Vehicle cabin") {
  Temperature: Float @vspec(element: SENSOR, fqn: "Vehicle.Cabin.Temperature", description: "Cabin temperature in °C") @range(min: -40, max: 100)
}

Start the server with:

vissv2server --vdm /path/to/vdm-files/

Then query a signal in the loaded tree:

{"action":"get","path":"Vehicle.Speed","requestId":"req-1"}
