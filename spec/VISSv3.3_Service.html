§10. Concurrent Service Invocation [NEW in 3.3]

Multiple clients may invoke the same procedure path concurrently. Each invocation is independent and is identified by a unique serviceId assigned by the server.

Normative rules:

  • The server MUST assign a distinct serviceId to each invocation regardless of whether another invocation of the same procedure is ONGOING.
  • Each invocation maintains its own state machine (UNKNOWN → ONGOING → terminal). The state of one invocation MUST NOT affect the state of another.
  • A Monitor request by path attaches to the most recently started ONGOING invocation for that path. If multiple ONGOING invocations exist, the server returns the one with the latest start timestamp.
  • A Cancel request by serviceId affects only the invocation or monitoring session identified by that serviceId.
  • Service processes are responsible for managing concurrency within a single procedure. The server makes no ordering or serialization guarantees between concurrent invocations.

§11. Service Timeout [NEW in 3.3]

An invocation that remains in the ONGOING state past its deadline SHALL be terminated by the server with status FAILED. The server MUST issue a FAILED terminal event to all monitoring sessions for that invocation and then discard the invocation state.

11.1 Default Timeout

The server implementation SHALL define a configurable default timeout. The reference implementation uses 30 seconds. If no timeout is specified in the request the default applies.

11.2 Per-Request Timeout

A client MAY include a "timeout" field in an invokeRequest to override the default. The value is a non-negative integer representing milliseconds. A value of zero means no timeout (the invocation runs until it terminates naturally or is cancelled).

§12. Server–Service Protocol [NEW in 3.3]

12.1 Overview

The VISSv3.2 specification defined the client–server interface in detail but left the mechanism by which the server communicates with a service process unspecified. VISSv3.3 defines a normative protocol for this internal interface.

The server acts as a TCP server. Service processes connect to a well-known port (ServiceRegPort, default 8300) and maintain a long-lived connection. All messages are UTF-8 JSON objects, one per line (newline-delimited). Messages flow in both directions on the same connection.

12.2 Wire Format

Messages are JSON objects terminated by a newline character (0x0A). The connection MUST use TCP. TLS is RECOMMENDED in production deployments. The server MUST accept connections on ServiceRegPort.

12.3 Registration

The service process sends a registration message immediately after connecting:

{
  "action": "register",
  "path": "VehicleService.Seating.MoveSeat",
  "signature": {
    "input":  {"SeatId": "string", "Position": "uint8"},
    "output": {"Position": "uint8"}
  }
}

The path field identifies the procedure node. The signature field describes the I/O parameters and their datatypes. The server uses the signature to dynamically construct a HIM service tree node for the procedure (see §17).

The server responds with an acknowledgement:

{"registered": true, "path": "VehicleService.Seating.MoveSeat"}

or an error:

{"registered": false, "reason": "missing path"}

A service process MUST NOT send further messages until it has received a successful ack.

12.4 Invocation Forwarding

When a client issues an invoke request for a procedure path that has a registered service process, the server forwards the invocation:

{
  "action": "invoke",
  "sessionId": "123456",
  "input": {"SeatId": "Row1-DriverSide", "Position": "40"}
}

The sessionId is the server-assigned invocation identifier (same value as the serviceId returned to the client). The service process uses this identifier in all subsequent progress messages for this invocation.

If no service process is registered for a procedure path, the server MAY either reject the invoke with a 503 error or maintain the invocation in ONGOING state and wait for a service process to register (implementation-defined). The reference implementation rejects immediately with status FAILED.

12.5 Progress Updates

The service process sends progress updates at any time after receiving an invoke message:

{"sessionId": "123456", "status": "ONGOING", "output": {"Position": "25"}}
{"sessionId": "123456", "status": "SUCCESSFUL", "output": {"Position": "40"}}

The output field is OPTIONAL for ONGOING updates and REQUIRED for terminal updates (SUCCESSFUL, CANCELED, FAILED) unless the procedure signature declares no output parameters.

The server MUST fan out monitoring events to all subscribed clients when it receives a progress update, subject to each session's filter settings (see VISSv3.2 §7).

Terminal status values (SUCCESSFUL, CANCELED, FAILED) MUST be sent at most once per sessionId. After sending a terminal update the service process MUST NOT send further messages for that sessionId.

12.6 Deregistration

A service process that wishes to gracefully disconnect sends:

{"action": "deregister"}

The server MUST then mark all ONGOING invocations for the procedure path as FAILED, issue the appropriate terminal events to monitoring clients, and remove the procedure from the HIM forest. The server closes the connection after processing the deregistration.

If the TCP connection is lost without a deregister message, the server MUST apply the same cleanup procedure as for explicit deregistration.

§13. Service Registration [NEW in 3.3]

A VISS server that supports VISSv3.3 MUST listen on ServiceRegPort (TCP, default 8300). This port number SHOULD be configurable. The server MUST accept connections from local and remote service processes, subject to any firewall or TLS policy.

A service process is considered registered when the server has sent a {"registered": true, …} ack. A registered service process MUST remain connected; the server treats a connection loss as an implicit deregister.

Multiple service processes MAY register for different procedure paths. Two service processes MUST NOT register for the same procedure path; the server SHALL reject a duplicate registration with {"registered": false, "reason": "path already registered"}.

§14. MQTT Topic Mapping for Service Operations [NEW in 3.3]

When service operations are transported over MQTT, the following topic conventions apply. The {VIN} segment is the vehicle identifier used as the MQTT namespace root. Path separators (dots) in procedure paths are replaced with forward slashes (/).

OperationDirectionTopic
Invoke requestClient → Broker/{VIN}/{path}/invoke
Invoke response / errorBroker → Client/{VIN}/{path}/invoke/response/{requestId}
Monitor requestClient → Broker/{VIN}/{path}/monitor
Monitor responseBroker → Client/{VIN}/{path}/monitor/response/{requestId}
Monitoring eventBroker → Client/{VIN}/{path}/monitoring/{serviceId}
Cancel requestClient → Broker/{VIN}/service/cancel
Cancel responseBroker → Client/{VIN}/service/cancel/response/{serviceId}
Discover requestClient → Broker/{VIN}/{path}/discover
Discover responseBroker → Client/{VIN}/{path}/discover/response/{requestId}

Example: an invoke request for VehicleService.Seating.MoveSeat on VIN WVGZZZ1KZ is published to topic:

/{WVGZZZ1KZ}/VehicleService/Seating/MoveSeat/invoke

The corresponding monitoring events are published to:

/{WVGZZZ1KZ}/VehicleService/Seating/MoveSeat/monitoring/123456

The MQTT QoS level for service messages SHOULD be QoS 1 (at least once) to prevent message loss. Clients MUST be prepared to handle duplicate deliveries.

§15. HTTP Method Binding for Service Operations [NEW in 3.3]

When service operations are transported over HTTP the following method bindings apply. This section extends VISSv3.2 §4 ("For HTTP the POST method is used for all request actions except for Service Discovery that uses the GET method."):

OperationHTTP MethodURL PathNotes
invokePOST/viss/service/{path}Body contains JSON invokeRequest (without action field).
monitorPOST/viss/service/{path}/monitorBody contains JSON monitorRequest.
cancelDELETE/viss/service/session/{serviceId}No request body required.
discoverGET/viss/service/{path}No request body; path may address a branch or procedure.

Since HTTP invoke and monitor requests can only receive the synchronous response (per VISSv3.2 §4), monitoring events for long-running services MUST be delivered via a separate mechanism such as Server-Sent Events (SSE) on a persistent GET connection, or via WebSocket upgrade.

The HTTP server MUST return status code 200 for successful responses and 400/500-series codes for error responses, with the JSON error body as defined in §8 of VISSv3.2.

§16. Service Software Development Kit (SDK) [NEW in 3.3]

An implementation of VISSv3.3 SHOULD provide an SDK that encapsulates the server–service protocol (§12) so that service authors do not need to implement raw TCP/JSON handling.

16.1 Normative SDK Interface

The reference Go SDK (package vissServiceSDK) defines the following interface. Implementations in other languages MUST provide equivalent semantics.

// Create a new service bound to path. serverAddr is the registration endpoint.
NewService(serverAddr, path string) *Service

// Declare a named input parameter and its datatype.
WithInput(name, datatype string) *Service

// Declare a named output parameter and its datatype.
WithOutput(name, datatype string) *Service

// Register the handler called for each incoming invocation.
OnInvoke(handler func(ctx *InvokeContext)) *Service

// Connect to the server, send the registration message, and return the
// registered service. Returns an error if connection or registration fails.
Register() (*Service, error)

// Block, reading and dispatching invoke messages. Returns when the
// connection is closed or Close() is called.
Run()

// Deregister and close the connection. Idempotent.
Close()

// InvokeContext methods:
//   SessionId  string                     — the invocation identifier
//   Input      map[string]interface{}     — the input parameters
//   ReportProgress(status string, output map[string]interface{}) error

16.2 SDK Usage Example

svc, err := vissServiceSDK.NewService("localhost:8300", "VehicleService.Seating.MoveSeat").
    WithInput("SeatId", "string").
    WithInput("Position", "uint8").
    WithOutput("Position", "uint8").
    OnInvoke(func(ctx *vissServiceSDK.InvokeContext) {
        target := ctx.Input["Position"]
        // ... move seat toward target ...
        ctx.ReportProgress("ONGOING",  map[string]interface{}{"Position": "25"})
        ctx.ReportProgress("SUCCESSFUL", map[string]interface{}{"Position": target})
    }).
    Register()
if err != nil { log.Fatal(err) }
defer svc.Close()
svc.Run() // blocks until server disconnects

§17. HIM Service Tree Configuration [NEW in 3.3]

A HIM service tree is any tree whose domain value has a right-most segment equal to Service (e.g., VehicleService.Car.Service). The server identifies service requests by checking this domain suffix via GetInfoType().

17.1 Binary Tree (Static Configuration)

A service tree MAY be provided as a pre-built binary file using the same format as VSS data trees. The viss.him entry follows the same syntax as data trees:

HIM.VehicleService:
type: direct
domain: VehicleService.Car.Service
version: 1.0
local: forest/vservice.v1.0.binary
description: Vehicle seating service tree (HIM service profile).

The binary file MUST be produced using a tree compiler that supports the HIM service profile node types: procedure, iostruct, property, symlink, and branch.

17.2 Dynamic Registration (Preferred for VISSv3.3)

Instead of a static binary file, a VISSv3.3 server MUST also support dynamic tree registration. When a service process connects and sends its signature (§12.3), the server:

  1. Parses the signature JSON to extract input and output parameter names and datatypes.
  2. Constructs an in-memory HIM node tree: a procedure node containing Input and Output iostruct nodes, each with property child nodes for each parameter.
  3. Registers the tree in the HIM forest under a domain whose right-most segment is Service, derived from the top-level segment of the registered path.
  4. Makes the tree immediately available for client requests (invoke, discover, etc.).

Dynamic registration removes the need to pre-build binary files for service trees. A viss.him entry is not required when dynamic registration is used.

The domain for a dynamically registered root named VehicleService MUST be VehicleService.Service so that GetInfoType() returns Service.

§18. Instantaneous Service Behaviour [NEW in 3.3]

VISSv3.2 §6.1 states that for services completing instantaneously, "the monitoring session is not initiated." VISSv3.3 clarifies the response content for this case:

  • The invokeSuccessResponse MUST have status set to SUCCESSFUL (not ONGOING).
  • The outdata field MUST contain the final output values.
  • The serviceId field MUST be omitted (no monitoring session was created).
  • No monitoring events will follow the synchronous response.

Whether a service is "instantaneous" is determined by whether the service process sends a terminal status (SUCCESSFUL, CANCELED, or FAILED) before the server sends the synchronous invokeSuccessResponse. In practice, if the service returns synchronously within the handler call, the server receives the terminal status before dispatching the response.

Example of an instantaneous invoke response:

{
  "action": "invoke",
  "path": "VehicleService.Seating.MoveSeat",
  "outdata": {"output": {"Position": "40"}, "ts": "2026-01-15T10:00:01Z"},
  "status": "SUCCESSFUL",
  "requestId": "8756",
  "ts": "2026-01-15T10:00:01Z"
}
