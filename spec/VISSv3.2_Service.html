Abstract

The Vehicle Information Service Specification (VISS) is a specification for access to vehicle information, including signals from sensors on control units within a vehicle's network, and services in the form of software applications that utilize vehicle signals for the service execution and are invoked by calling a procedure having input and output parameters. This information is exposed through a hierarchical, tree-like taxonomy as defined in the COVESA Hierarchical Information Model (HIM), and is provided in JSON format. A server implementing VISS may be hosted within the vehicle or on external platforms.

VISS enables a broad range of use cases, including predictive maintenance, usage-based insurance, fleet management, and real-time driver assistance services. In the context of artificial intelligence, VISS serves as a critical data access layer, providing high-quality, real-time vehicle data that can be utilized by AI algorithms for tasks such as anomaly detection, driver behavior analysis, energy optimization, and contextual decision-making.

The first version of VISS, which has been implemented and deployed in production vehicles, supported only WebSocket as a transport protocol.
The version 2.0 is generalized to work across also the HTTP and MQTT transport protocols to improve the support for different use cases. It also contains improved subscription capabilities and added an access control mechanism.
The version 3.0 included the gRPC transport protocol but also simplified addition of other transport protocols by placing the normative requirement on the message payload only, and not also on what transport protocol to use. Several new features were added such as file transfer, payload encoding, and improved server capabilities representation.
The version 3.1 replaced the VSS rule set for how to define the content of a tree with the HIM data profile rule set and included support for the Unix domain socket transport protocol.
This specification, version 3.2, extends the usage of the HIM data model to also include the HIM service profile.

There are four parts to this specification, [[CORE]], [[TRANSPORT]], [[PAYLOAD ENCODING]], and SERVICES. This document, the VISS version 3.2 SERVICES specification, describes the service related parts of the interface. The companion specification [[CORE]] describes the messaging layer, [[TRANSPORT]] describes the deviations from the CORE specification that are used by some transport protocols, and [[PAYLOAD ENCODING]] describes payload encodings that are used in some cases.

This specification describes the service related parts of the VISS interface. Services are typically realized as software components that are deployed as shown in the figure below.

The service is the logical end point of the client request; the service implementation then uses signals in the vehicle signal domain to execute the service. The signal access can either be directly to the native signals or it can use the VISS signal abstraction; the latter may enable the service to be interoperable on different vehicle models and brands.

The VISS Service profile follows the model from VISS Vehicledata profile in that the data describing the service signature (procedure name, input, output) is handled as data in the interface. Any changes to the service signature will thus not require any changes of the interface. This is true for all changes all the way to a complete switch to another service domain tree.

As well as sections marked as non-normative, all authoring guidelines, diagrams, examples, and notes in this specification are non-normative. Everything else in this specification is normative.

The acronym 'VISSv3.2' is used to refer to this document, the VISS version 3.2 specification.
The acronym 'HIM' is used to refer to the 'Hierarchical Information Model' which is hosted by COVESA.
The acronym 'VSS' is used to refer to the 'Vehicle Signal Specification' which is hosted by COVESA.
The term 'WebSocket' when used in this specification is as defined in the W3C WebSocket API and [[RFC6455]], the WebSocket Protocol.

1. Service Definition

A service is in this specification defined as a software application that utilizes vehicle signals for the service execution and a client invokes it by calling a procedure having input and output parameters. A service is defined by a service signature as shown below.

ProcedureName(input1: datatype1, ..., inputN: datatypeN): (outputA: datatypeA, ..., outputM: datatypeM)

where ProcedureName is the name of the procedure, inputX is the X:th input parameter, datatypeX is the datatype of the X:th input parameter, outputY is the Y:th output parameter, datatypeY is the datatype of the Y:th output parameter.
The procedure shall have zero or more input parameters and one or more output parameters.
The signature components are mapped to nodes in a HIM service tree as shown below.

  • ProcedureName: The procedure name is mapped to a node with the name ProcedureName and the node type 'procedure'.
  • inputX: An input parameter is mapped to a node with the name InputX and the node type 'property' or 'symlink'. This node must be a child to a node with the name Input and the node type 'iostruct', which in its turn must be a child to the node with the name ProcedureName.
  • outputY: An output parameter is mapped to a node with the name OutputY and the node type 'property' or 'symlink'. This node must be a child to a node with the name Output and the node type 'iostruct', which in its turn must be a child to the node with the name ProcedureName.

The described tree structure is shown in the figure below.

The service data model is defined by the HIM service profile.

2. Service States

The status of a service is a stateful parameter and shall be preserved by the server. It must be one of the following states:

  • UNKNOWN: The status of the service is unknown.
  • ONGOING: The service has been invoked but has not yet reached a successful termination, or terminated due to being canceled or ending in failure.
  • SUCCESSFUL: The service is not ONGOING and the latest invocation of the service terminated successfully.
  • CANCELED: The service is not ONGOING and the latest invocation of the service terminated due to being canceled.
  • FAILED: The service is not ONGOING and the latest invocation of the service terminated due to a failure.

The Status string datatype is restricted to the allowed values from the list above. The Status parameter shall be included in all response and event messages, set to the current value. At server startup it may not have information about the status of a service. It shall then be set to UNKNOWN. Input and output data in response or event messages are undefined if the status is set to UNKNOWN or FAILED.

3. Actions

The Action parameter shall be included in all request, response and event messages. For transport protocols with methods that do not define a payload the 'action' parameter shall be omitted.
The different requests shall use the action values shown in the list below. The same action value shall be used in the response message.

  • Invoke: "action": "invoke"
  • Monitor: "action": "monitor"
  • Cancel: "action": "cancel"
  • Service Discovery: "action": "discover"

Event messages shall have the following action value:

  • Event: "action": "monitoring"

4. Data Formats

The VISS signal profile defines a "data" object for the JSON representation of the output signal(s). This data represents the current value of the signal(s) together with capture time stamp(s).
The VISS service profile data model has separate definitions for input and output data which then each are correspondingly defined below.
The definition of the service input data is shown below.

"indata": {"input": {input-params}, "ts": "ddtt"}

where "input-params" is the JSON representation of the service input parameters with their corresponding values and "ddtt" is the timestamp value associated with the server reception of the input parameters.
The definition of the service output data is shown below.

"outdata": {"output": {output-params}, "ts": "ddtt"}

where "output-params" is the JSON representation of the service output parameters with their corresponding values and "ddtt" is the timestamp value associated with the capture of the output parameters.

  • For HTTP the POST method is used for all request actions except for Service Discovery that uses the GET method.
  • The Invoke and the Monitor requests will if sent over HTTP only return the response message.

5. Service Interface

The service interface builds on the syntax and concepts used in the CORE:Data Interface for access to vehicle signals. The differences are listed below.

  • The service interface must only be used on trees that have a 'domain' metadata value with a right-most segment named 'Service', as defined in the HIM configuration file (Appendix B).
  • The available methods are: invoke / monitor / cancel / discover.
  • A request can only invoke one service.
  • All response, event and error messages shall include a Status parameter.
  • The requests defined in the service interface shall only be issued using paths addressing nodes of node type 'procedure', except for service discovery requests that may use a path addressing nodes of type 'branch'.
  • The available filter variants are: timebased / status / all / none.
  • If an invoke or monitor request leads to that an event session is started then it is automatically terminated when the service is successfully terminated, or terminated due to being cancelled or ending in failure. The server must then issue an event or error message with the Status set appropriately.
  • The client that invoked a service and has access to the serviceId of the ONGOING service may terminate the service at any point by calling Cancel.
  • A client monitoring an ONGOING service without having invoked it cannot terminate the service by calling Cancel, but the Cancel call will terminate its monitoring session.
  • A monitor request returns the latest input values that have been used to invoke the service in the response. If the response Status has the value ONGOING then events will be issued until the Status value is changed to either SUCCESSFUL, CANCELED or FAILED.

A client can apply the following types of operations on a service:

  • Invoke: A request to execute a service.
  • Monitor: A request to get monitoring data of a service.
  • Cancel: A request to cancel the session started by a previous invoke or monitor request.
  • Discovery: A request to get metadata describing one or more services.

The service signature is declared in the tree following the HIM Service profile syntax; a client request must conform to the service signature. A service shall have zero or more input parameters, and one or more output parameters. If the request does not conform to the signature the server shall reply with an error message and shall not begin execution of the service. The server must also issue an error message if it during the service execution enters into a failure state.

The time duration that a service requires for its execution may be negligible in terms of the total time between that the request was issued and that a response was received. In such cases the service is not expected to issue any asynchronous event messages to report its execution progress. Services that have a non-negligible time duration shall issue event messages to report its execution progress. The rate of event messages is an implementation decision, but the service must as a minimum issue an event at the successful termination of the execution, or if it enters into a failure state.

To realize these operations the methods described in the following chapters are available.

6.2 Monitor

Purpose: Initiate a monitoring session of a service.
If the service execution is ongoing and the filter variant is not set to "none" then a monitoring session is initiated, else only the synchronous response will be issued.
The monitor request starts a monitoring session of the service. The monitoring session is directly terminated if the service is not in the ONGOING state, initiated by a previous request by some client. A synchronous response is sent back to the client, which is then followed by asynchronous monitoring event messages. The response contains the input parameters that were used when the service was initiated. In the case of a service that completes its execution instantaneously the monitoring session is not initiated. The monitoring session is terminated when the service terminates successfully, or enters into a failure state.
Request arguments, of which path and filter are mandatory:

  • path: The path to a node with the node type procedure.
  • filter: Additional parameters configuring the type of monitoring.
  • authorization: The authorization token.

  • path: The path to the service.
  • indata: The input parameters as specified by the service signature. The input parameters contain the values from the latest invocation of the service.
  • outdata: The service output parameters as specified in the service signature.
  • status: The execution Status.
  • serviceId: A reference Id for the monitoring session.

Purpose: Cancels either an ongoing service or a monitoring session.

  • serviceId: The Id of an ongoing session.

7. Filters

The service filter is used to configure the event message triggering during the service monitoring session. The rate at which a service updates its output parameters is implementation specific and it is typically different for different services. The service implementation shall update its Status parameter directly when the state of the service changes. The filter settings do not affect this "native" rate of the service implementation; it can only reduce the rate of events being issued to the client. If the filter setting requests a higher rate than the native rate it will get events at the native rate. Services that do not issue any events will ignore the filter and act as if the variant is set to "none". Information on what is the native rate should be available in documentation of the service.
The syntax follows the Data profile filter syntax with variant and parameter keys as shown below.

"filter": {"variant": "name", "parameter": {"param-name": "param-value"}}

There is one difference in that the "parameter" key-value pair is optional as it is not used in some variants. The supported variants are defined in the following.

7.1 Timebased Filter

This filter variant issues events with a fixed time period between the events. The syntax is shown below.

"filter": {"variant": "timebased", "parameter": {"period": "param-value"}}

where "param-value" is an integer and represents the period time in milliseconds. Events will also be issued if the Status parameter changes.

7.2 Status Filter

This filter variant issues events when the Status parameter changes. The syntax is shown below.

"filter": {"variant": "status"}

The "parameter" key-value pair shall not be present.

7.3 All Filter

This filter variant issues events when either the Status parameter changes, or any output parameter changes. This is the same as the service native rate. The syntax is shown below.

"filter": {"variant": "all"}

The "parameter" key-value pair shall not be present.

7.4 None Filter

This filter variant does not issue any events. The syntax is shown below.

"filter": {"variant": "none"}

The "parameter" key-value pair shall not be present.

8. Error Messages

A service error message shall contain error information as defined in CORE:Error Information and the service Status parameter.

The error message has the same principal format for all the different requests. The difference between an error response message and an error event message is that in the former the key name 'requestId' is used, in the latter the key name 'serviceId' is used.

9. Examples

The service examples are shown using JSON, the primary payload format. The payloads include the parameters 'action' and 'requestId' which are needed if transported over e.g. WebSocket, or MQTT. See Section 3: Actions for the 'action' values.

9.1 Invoke Example

The invoke request initiates the service execution and starts a monitoring session of the service execution. A synchronous response is sent back to the client, which is then followed by asynchronous monitoring event messages. In the case of a service that completes its execution instantaneously the monitoring session is not initiated. If the execution has a longer duration then at least one event message will be issued, when the service terminates successfully, or enters into a failure state.

Request:

{
    "action": "invoke",
    "path": "VehicleService.Seating.MoveSeat",
    "input": {"SeatId": {"Row":"Row1", "Column": "DriverSide"}, "MovementType": "longitudinal", "Position": "40"},
    "filter": {"variant": "timebased", "parameter": {"period": "250"}},
    "requestId": "8756"
}

Successful response:

{
    "action": "invoke",
    "path": "VehicleService.Seating.MoveSeat",
    "outdata": {"output": {"Position": "20"}, "ts": "2025-04-15T13:37:04Z"},
    "serviceId": "123",
    "status": "ONGOING",
    "requestId": "8756",
    "ts":"2025-04-15T13:37:05Z"
}

Event message:

{
    "action": "monitoring",
    "path": "VehicleService.Seating.MoveSeat",
    "outdata": {"output": {"Position": "30"}, "ts": "2025-04-15T13:37:05Z"},
    "serviceId": "123",
    "status": "ONGOING",
    "ts":"2025-04-15T13:37:06Z"
}

9.2 Monitor Example

For services that do not complete their execution instantaneously an invoke or monitor request may lead to that a monitoring session is started. This session can be canceled by the client that started it.

Request:

    "filter": {"variant": "all"},

    "serviceId": "123",
    "status": "ONGOING",
    "requestId": "8756",
    "ts":"2025-04-15T13:37:05Z"
}

Event message:

{
    "action": "monitoring",
    "path": "VehicleService.Seating.MoveSeat",

Appendix B: HIM Configuration for Service Trees

A server determines whether a given HIM tree is a service tree (and therefore eligible for the service interface operations) by inspecting the domain metadata value in its HIM forest configuration file (viss.him). A tree is a service tree if and only if the right-most dot-separated segment of its domain value is the literal string Service.

Example viss.him entry for a service tree:

HIM.VehicleService:
type: direct
domain: Vehicle.Car.Service
version: 1.0
local: forest/vservice.v1.0.binary
description: Vehicle service tree (HIM service profile).

The server exposes the right-most segment via its internal GetInfoType() function. Any tree for which GetInfoType() returns "Service" is routed through the service interface described in this specification; all other trees are handled by the standard data interface.

The HIM configuration file format, including the syntax for type, domain, version, local, and public keys, is defined in the VISSv3.1 [[CORE]] specification.
